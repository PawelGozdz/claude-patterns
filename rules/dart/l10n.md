# Flutter Localization Rules

## CRITICAL: All user-facing strings MUST use ARB localization

NEVER hardcode Polish or English strings in presentation layer files.
Always use `context.l10n.keyName` or `l10n.keyName` pattern.

### Required pattern

```dart
// CORRECT
Text(context.l10n.saveButton)
Text(l10n.profileEditTitle)

// WRONG - hardcoded Polish
Text('Zapisz')
Text('Edytuj profil')
```

### Where localization is required

- `lib/features/*/presentation/screens/**/*.dart`
- `lib/features/*/presentation/widgets/**/*.dart`
- Any file that displays text to users

### Where localization is NOT required

- Domain layer (use string keys, resolve in presentation)
- Data layer (API values, JSON keys)
- Test files
- Route paths, widget keys, enum values

### How to add new localized strings

1. Add key+value to `lib/l10n/app_pl.arb` (Polish - primary)
2. Add key+English translation to `lib/l10n/app_en.arb`
3. Add getter delegation to `lib/core/localization/app_localizations.dart` wrapper
4. Run `flutter gen-l10n` to regenerate
5. Use `context.l10n.keyName` in presentation code

### Key naming convention

- lowerCamelCase: `featureName_screenContext_element`
- Examples: `authLoginTitle`, `profileEditSaveButton`, `localSharesEmptyState`

### For domain layer strings

Domain layer cannot access `context.l10n`. Use key-based approach:
```dart
// Domain: return string keys
class PasswordRequirement {
  final String key; // e.g., 'minLength', 'uppercase'
}

// Presentation: resolve keys to localized strings
extension PasswordRequirementL10n on PasswordRequirement {
  String resolve(AppLocalizations l10n) => _keyMap[key] ?? key;
}
```

### Interpolation

Use ARB placeholder syntax for dynamic values:
```json
{
  "welcomeUser": "Witaj, {name}!",
  "@welcomeUser": { "placeholders": { "name": { "type": "String" } } }
}
```

### Enforced by

- `flutter-hooks.json` l10n section (detects hardcoded Polish in presentation)
- Code review checks
- `check-flutter-imports.js` Stop hook
