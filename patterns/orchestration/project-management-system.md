# Pattern: Project Management System

**Layer**: Orchestration  
**Status**: production  
**Works with**: any project using claude-code  
**Agents**: `tech-lead`, `product-owner`  
**Skills**: `/pulse`, `/pm-status`, `/task-health`, `/tech-debt`, `/sprint`

---

## What This Is

A lightweight, file-based project management system that gives any claude-code
project a **living team** — two advisory agents (technical and business lenses)
sharing a common state file, triggered by task changes and periodic reviews.

The system is intentionally minimal: no external tools, no databases, no
dashboards. Just markdown files, YAML frontmatter, and claude agents that know
where to look.

**Philosophy**: A team without shared memory is a collection of individuals.
This system's core innovation is `TEAM-STATE.md` — a single file that all
agents read first and write to after analysis. It creates continuity across
sessions that can last days or weeks.

---

## The Three Components

### 1. Shared Brain: `TEAM-STATE.md`

The heartbeat of the system. Lives in `project-orchestration/TEAM-STATE.md`.
Every agent reads it at the start. The `/pulse` skill updates it after each
team sync.

```markdown
# Team State — {PROJECT}
*Last sync: {DATE} by @{AGENT}*

## 🎯 Sprint Focus
{Current sprint or phase}

## 🔴 Critical Now
{P0/P1 blocked or overdue tasks — updated by @tech-lead}

## ⚙️ Technical Pulse
**Debt**: 🟡 MEDIUM | Major: 2 | Minor: 5
**Blocked chains**: 1 | **Stale (>14d)**: TS-XXX, TS-YYY
[{date}] @tech-lead: note

## 💼 Business Pulse
**Next milestone**: {milestone} — est. {weeks}
**Unvalidated**: 2 features | **Mobile risks**: 3 tasks
[{date}] @product-owner: note

## 📝 Team Notes
[{date}] @agent: insight
```

### 2. Task Folder: `project-orchestration/`

```
project-orchestration/
├── TEAM-STATE.md          ← shared brain (git-tracked, auto-updated)
├── KANBAN.md              ← board view (auto-generated by /pulse)
├── TECH-DEBT.md           ← debt register (updated by @tech-lead)
├── README.md              ← system explanation
│
├── tasks/                 ← active tasks (YAML frontmatter, see schema)
├── completed-tasks/       ← archive (immutable once moved)
└── _archive/              ← old docs, analyses, historical content
```

### 3. Agent Pair

| Agent | Lens | Reads | Writes |
|-------|------|-------|--------|
| `tech-lead` | Technical | tasks/, git log, BUSINESS_RULES.yaml | TEAM-STATE.md (Technical Pulse) |
| `product-owner` | Business | tasks/, product docs, validation evidence | TEAM-STATE.md (Business Pulse) |

Both agents are **advisory** — they analyze and output insights, they do not
implement code. Projects can override `product-owner` with a version that has
VETO power (see LocalHero example below).

## Task YAML Schema

All tasks in `tasks/` use YAML frontmatter. Minimum required fields:

```yaml
---
id: TS-XXX
title: 'Descriptive task title'
status: planned|ready|in-progress|blocked|done|deferred
priority: P0|P1|P2|P3
story_points: 5
created_date: YYYY-MM-DD
updated_date: YYYY-MM-DD        # MUST update when task changes
assignee: '@agent-or-person'
labels: [backend, auth, testing]
---
```

### Extended Fields (enable system intelligence)

```yaml
due_date: YYYY-MM-DD            # → overdue detection by @tech-lead
mobile_impact: none|low|medium|high  # → UX risk flag for @product-owner
tech_debt: none|minor|major     # → aggregated in TECH-DEBT.md
dependencies: [TS-AAA, TS-BBB] # → upstream blockers
blocks: [TS-CCC]               # → downstream impact (reverse deps)
story_id: US-XXX               # → business traceability
```

### Priority Scale

| Priority | Meaning | SLA |
|----------|---------|-----|
| P0 | Blocks everything / security / production incident | Act today |
| P1 | Blocks milestone or other P0/P1 tasks | Act this sprint |
| P2 | Important but not blocking | Next sprint |
| P3 | Nice to have / low urgency | When capacity allows |

## The Trigger Mechanism

The system is **event-driven + periodic**, not session-start dependent.
This matters for long-lived tmux sessions.

### Event: Task File Changed (PostToolUse Hook)

When any file in `project-orchestration/` is written or edited, the
`pm-task-check.js` hook fires. It:
1. Checks if the file is a task file (YAML frontmatter)
2. If yes: scans all active tasks and prints a compact briefing
3. Claude sees this in context and can surface relevant insights

Hook output example:
```
[PM] 74 active | P0: 3 | P1: 12 | Blocked: 5 | Stale: 8
[PM] Overdue: TS-AUTH-003 (due 2026-03-15, 18d ago)
[PM] Mobile HIGH (pending): TS-GEO-013, TS-AUTH-MOBILE-001
[PM] Debt: 🔴 3 major, 7 minor
```

### Periodic: `/pulse` Skill

The main team sync. Runs both agents in sequence:
1. `@tech-lead` analyzes technical state → outputs Technical Pulse block
2. `@product-owner` analyzes business state → outputs Business Pulse block
3. Claude updates `TEAM-STATE.md` with both outputs
4. Claude regenerates `KANBAN.md` from current task state

Run manually (recommended: start of each working day) or via cron.

### On-Demand Skills

| Skill | When | Cost |
|-------|------|------|
| `/pm-status` | Quick check — reads TEAM-STATE.md only | ~$0 (no agent) |
| `/pulse` | Full sync — runs both agents | ~$0.10 |
| `/task-tidy` | Housekeeping — move done, fix fields, validate YAML | ~$0.03 |
| `/task-health` | Deep audit — @tech-lead analyzes all deps | ~$0.05 |
| `/tech-debt` | Debt report — @tech-lead focuses on debt | ~$0.05 |
| `/sprint` | Sprint planning — both agents, interactive | ~$0.20 |

## Setup for a New Project

### 1. Copy the Template

```bash
cp -r /path/to/claude-patterns/templates/project-orchestration ./project-orchestration
```

### 2. Symlink Agents (if not already global)

```bash
# These should already be in ~/.claude/agents/ via claude-patterns symlink
# Verify:
ls ~/.claude/agents/ | grep -E "tech-lead|product-owner"
```

### 3. Add Hook to `.claude/settings.json`

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "node /path/to/claude-patterns/hooks/pm-task-check.js"
          }
        ],
        "description": "PM health check when task files change"
      }
    ]
  }
}
```

### 4. Initialize TEAM-STATE.md

Edit `project-orchestration/TEAM-STATE.md` with your project's initial state,
then run `/pulse` to let both agents do their first analysis.

### 5. Optional: Project-Specific Product Owner Override

If your project needs VETO power on `product-owner`, create
`.claude/agents/product-owner.md` in the project that extends the universal
agent with project-specific business rules and VETO authority.

## Anti-Patterns

**❌ Skipping `updated_date`**: Stale detection breaks. Every task edit must
update this field — make it a habit or add it to commit hooks.

**❌ Tasks without `priority`**: @tech-lead can't determine criticality.
Default to P2 if unsure, but always set it.

**❌ Writing directly to TEAM-STATE.md**: Use `/pulse` or agent output.
Manual edits lose the agent perspective and timestamp trail.

**❌ Keeping completed tasks in `tasks/`**: Move them to `completed-tasks/`
immediately. The health metrics are based on `tasks/` count.

**❌ Growing `_archive/`**: Everything in `project-orchestration/` that isn't
a task or a state file belongs in `_archive/`. Keep the root clean.

---

## Project-Specific Override: LocalHero Example

LocalHero overrides `product-owner` with `localhero-product-owner` (in
`.claude/agents/`) which adds:
- VETO power (blocks implementation without validated business value)
- B2C/B2B/B2G segment mapping specific to Starachowice context
- Mom Test evidence requirement
- References to LocalHero business docs and grant validation files

```yaml
# .claude/agents/localhero-product-owner.md
---
name: localhero-product-owner
description: |
  Product Owner for LocalHero — extends universal product-owner with
  VETO POWER and LocalHero-specific B2C/B2B/B2G validation.
  Supersedes customer-value-guardian.
...
---
```

This pattern (universal agent + project override) allows the system to be
both reusable and project-specific.

## File Conventions

| File | Owner | Update Frequency |
|------|-------|-----------------|
| `TEAM-STATE.md` | Agents (via /pulse) | Each /pulse run |
| `KANBAN.md` | Claude (via /pulse skill) | Each /pulse run |
| `TECH-DEBT.md` | @tech-lead | Each /task-health or /pulse |
| `tasks/*.md` | Human + agents | On task change |
| `completed-tasks/` | Human | On task completion |
| `_archive/` | Human (one-time cleanup) | Rarely |
